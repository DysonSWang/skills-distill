## 一、激活触发条件评审

### 当前触发词清单

| 类别 | 触发词 |
|------|--------|
| 完整意图 | 「小红书运营」「小红书变现」「起号涨粉」 |
| 动作指令 | 「做小红书」「小红书怎么做」 |
| 问题症状 | 「这条笔记怎么改」「账号为什么不涨粉」 |
| 专家视角 | 「用运营视角」「运营专家会怎么看」 |

### 遗漏的高概率真实触发词

**症状类（用户说问题而非方案）**：
- ❌ 「涨粉慢」「涨不动」「不涨粉」
- ❌ 「爆不了」「发什么都没人看」「没人点赞」
- ❌ 「笔记没人看」「推荐量低」
- ❌ 「账号做死了吗」「账号废了」
- ❌ 「怎么上热门」「怎么爆」
- ❌ 「接广告」「怎么赚钱」（变现意图但表达随意）

**口语化/懒表述类**：
- ❌ 「小红书咋做」「小红书怎么弄」
- ❌ 「帮我看看账号」「帮我看看笔记」
- ❌ 「有什么建议」（无上下文但可能触发Skill）

### 覆盖盲区分析

当前触发词设计假设用户「知道自己在问运营问题」，但真实用户场景恰恰相反：

```
真实用户路径：描述症状 → 期待诊断 → 才意识到是运营问题
```

例：「我发了好几十篇了怎么还是几百个粉丝」——这句话包含「粉丝」但不含「涨粉」，当前触发词可能无法激活。

### 建议补充的触发词

```
涨粉慢、涨不动、不涨粉、爆不了、发什么都没人看、
推荐量低、账号废了、怎么做都火不起来、笔记没人看、
怎么上热门、怎么爆、小红书怎么弄、帮我看看账号
```

## 二、角色扮演规则可操作性评审

### 规则清单与缺失

| 规则类型 | 现状 | 缺失评估 |
|---------|------|---------|
| 开场规范 | ✅ 有（「我帮你分析」「按这个思路」开场） | — |
| 结论风格 | ✅ 直接给结论，不铺垫 | — |
| 数据处理 | ✅ 先查再做判断 | — |
| 免责声明 | ✅ 仅首次激活说一次 | — |
| 退出规则 | ✅ 「退出」「切回正常」 | — |
| **问题路由** | ❌ 无 | 遇到小红书运营以外的问题如何处理？ |
| **频率约束** | ❌ 无 | 每次分析说多少算合适？说太久vs太短？ |
| **失败预防** | ❌ 仅有退出规则，无退出条件 | 什么情况下应该主动说「这个我帮不了」？ |

### 问题路由（关键缺失）

Skill无任何「边界感」说明。当用户问以下问题时，Skill会如何响应？

- 「帮我写一篇小红书文案」（内容生产，而非策略分析）
- 「小红书和抖音有什么区别」（平台对比，超出运营视角）
- 「我被限流了怎么办」（具体技术问题，超出方法论范畴）
- 「你给我推荐几个赛道」（主观推荐，缺乏用户画像）

**风险**：Skill可能被用于回答它并不擅长的任务类型，导致输出质量不稳定。

### 频率约束（缺失）

Skill文档长，内容深度高，但没有告诉Agent：
- 每次响应应该覆盖多少模型/启发式？
- 用户问一个具体问题时，应该调用几个工具/模型来分析？
- 「说完就走」vs「系统讲解」如何取舍？

当前设计隐含「按需调用」，但缺乏执行标尺。

### 失败预防（部分缺失）

现有规则只有「用户主动退出」，缺少「Skill主动退出」的条件：

以下情况应主动退回到正常模式，而非强行以小红书专家身份硬答：
- 用户明确要求写文案（非分析，非顾问）
- 用户问的是具体技术操作（如何注册、如何开通某个功能）
- 用户表达的是投诉/情绪而非求教（平台误判了我的内容）

## 三、缺失的关键信息识别

### 小红书运营最有价值的3个问题

| # | 问题 | Skill能否回答 |
|---|------|-------------|
| 1 | 「账号定位：我该做什么赛道？」 | ✅ 部分（三圈交集模型）但缺乏实操路径 |
| 2 | 「为什么我的笔记不爆？如何改进？」 | ✅ 强（爆款四要素+决策启发式） |
| 3 | 「如何持续变现？什么时候能接广告？」 | ✅ 部分（变现时机门槛启发式） |

**结论**：核心高频问题覆盖较好，但问题1的实操性偏弱（没有提供定位诊断的具体步骤）。

### Skill最难回答的问题

| 问题类型 | 典型问法 | 难度原因 |
|---------|---------|---------|
| 爆款预测 | 「我这篇能爆吗？」 | 算法黑盒，无法预测，只能评估潜力 |
| 内容生产 | 「帮我写标题/封面文案」 | Skill是顾问不是写手，边界模糊 |
| 具体账号诊断 | 「你看我这个号有什么问题」 | 缺乏具体数据输入，分析流于泛化 |
| 竞品对比 | 「我和XX博主差在哪」 | 缺乏对标博主的详细数据 |

**最危险的问题**：用户扔一个笔记链接问「帮我看看怎么改」——Skill可以分析（如果能看的话），但如果没有具体内容上下文，分析会变成「套模板的泛泛而谈」，反而降低用户信任。

## 四、具体文本改动建议

### 改动1：Frontmatter触发词扩充（高优先级）

**文件位置**：第7-8行

**原文**：
```
当用户提到「小红书运营」「做小红书」「起号涨粉」「爆款笔记」「小红书变现」「用运营视角」「运营专家会怎么看」时使用。
即使用户只是说「这条笔记怎么改」「账号为什么不涨粉」「小红书怎么做」也应触发。
```

**建议改为**：
```
当用户提到以下任一场景时使用：
- 完整意图：「小红书运营」「小红书变现」「起号涨粉」「做账号」
- 问题症状：「涨粉慢」「涨不动」「爆不了」「笔记没人看」「推荐量低」「账号废了」「怎么上热门」
- 操作需求：「这条笔记怎么改」「账号为什么不涨粉」「小红书怎么做」「帮我看看账号」
- 专家视角：「用运营视角」「运营专家会怎么看」

注：用户说「帮我写文案」「帮我取标题」时，仅在明确要求分析策略时激活，不主动代写内容。
```

**理由**：症状类触发词是真实高频场景，当前设计严重遗漏；增加边界说明可防止Skill被滥用为文案代写工具。

### 改动2：激活规则增加「问题路由」与「主动退出条件」（高优先级）

**文件位置**：第16-25行「激活规则」章节

**原文**：
```
## 激活规则

**此Skill激活后，以小红书运营专家的身份直接给出判断和建议。**

- 用「我帮你分析」「按这个思路」开场
- 直接给结论，不铺垫
- 遇到需要数据的问题，先查再做判断
- **免责声明仅首次激活时说一次**（如「我按小红书运营专家的框架帮你分析，基于公开方法论总结」），后续不再重复

**退出规则**：用户说「退出」「切回正常」时恢复正常模式
```

**建议改为**：
```
## 激活规则

**此Skill激活后，以小红书运营专家的身份直接给出判断和建议。**

- 用「我帮你分析」「按这个思路」开场
- 直接给结论，不铺垫
- 遇到需要数据的问题，先查再做判断
- **免责声明仅首次激活时说一次**（如「我按小红书运营专家的框架帮你分析，基于公开方法论总结」），后续不再重复

**边界说明（问题路由）**：
以下情况不激活或主动退出：
- 用户要求「帮我写/取标题」「帮我写文案」→ 内容生产超出顾问范畴，建议说「我是策略顾问，具体文案可以告诉我主题后帮你分析思路」
- 用户问平台具体功能/注册流程 → 超出运营方法论范畴，建议说「具体操作问题建议直接查官方指引」
- 问题明显与小红书无关 → 主动退出

**失败预防（主动退出条件）**：
- 连续3次无法基于已有信息给出具体建议时，说明局限并退出
- 用户表达强烈不满（「你说的太泛了」「没用」）时，承认局限并建议补充具体背景

**退出规则**：用户说「退出」「切回正常」时恢复正常模式
```

**理由**：当前激活规则没有路由边界，导致Skill可能被用于回答不擅长的问题；增加失败预防可以保护Skill输出质量的下限。
